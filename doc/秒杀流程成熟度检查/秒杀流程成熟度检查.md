# 秒杀流程成熟度检查报告

> 检查时间：基于当前代码库
> 检查范围：从请求入口到订单创建、超时关单的完整链路

---

## 一、流程总览

```
用户请求 SeckillVoucher(voucherId, userId)
    │
    ├─ 1. querySeckillVoucherById (Redis 缓存 / MySQL 回填)
    ├─ 2. 校验时间窗口 (BeginTime ~ EndTime)
    ├─ 3. 生成 orderId (Redis INCR)
    ├─ 4. 发送事务消息 (RocketMQ 半消息)
    │       └─ ExecuteLocalTransaction: Lua 预减 Redis + 防重复
    │       └─ Commit → 消息投递到消费者
    │
    └─ 返回「排队中」

消费者 processOrder
    ├─ 5. 分布式锁 (lock:order:userId)
    ├─ 6. createVoucherOrder (MySQL 事务: 校验重复 + DecrStock + Create)
    ├─ 7. 发送订单超时延迟消息 (30min)
    └─ Unlock

30 分钟后 HandleOrderTimeout
    ├─ 8. 若 status==NOTPAYED → 关单 + IncrStock(MySQL) + Lua 回滚 Redis
    └─ 幂等：已支付/已取消则跳过
```

## 二、已实现的优化 ✅

| 优化项 | 实现位置 | 说明 |
|--------|----------|------|
| **Redis 预减库存** | `voucher_script.lua` | Lua 原子执行：库存判断 + 防重复 + 扣减 + SADD |
| **事务消息** | `rocketmq.go` | 半消息 → Lua → Commit/Rollback，保证「扣 Redis」与「发消息」原子性 |
| **异步削峰** | RocketMQ 消费者 | 请求快速返回，订单异步写库，削峰填谷 |
| **秒杀优惠券缓存** | `querySeckillVoucherById` | Redis 缓存 5 分钟，减轻 MySQL 压力 |
| **Redis 库存预热** | `voucher_logic.AddSeckillVoucher` | 创建秒杀券时异步写入 `seckill:stock:voucherId` |
| **分布式锁** | `processOrder` | `lock:order:userId` + 看门狗续期，防同一用户并发消费 |
| **MySQL 乐观锁** | `DecrStock` | `WHERE stock > 0` 防止超卖 |
| **订单 ID 生成** | `redisx.RedisWork.NextId` | Redis INCR 生成，避免单点 |
| **订单超时关单** | 延迟消息 + `HandleOrderTimeout` | 30 分钟未支付自动关单 + 回滚 Redis/MySQL |
| **回查机制** | `CheckLocalTransaction` | Producer 崩溃时根据 Redis 状态判断 Commit/Rollback |

## 三、待完善 / 可优化点

### 3.1 高优先级

| 项目 | 现状 | 建议 |
|------|------|------|
| **秒杀优惠券 ID 布隆过滤器** | ✅ 已实现 | 启动时预热 `bf:seckill-voucher`，SeckillVoucher 前校验；AddSeckillVoucher 时同步加入。 |
| **限流** | ✅ 已实现 | 使用 `golang.org/x/time/rate`，秒杀接口限流（默认 1000 QPS，可配 `SECKILL_RATE_LIMIT`/`SECKILL_RATE_BURST`），超限返回 429。 |
| **唯一索引兜底** | ✅ 已实现 | `VoucherOrder` 增加 `uniqueIndex:idx_user_voucher`，GORM AutoMigrate 自动创建。 |

### 3.2 中优先级

| 项目 | 现状 | 建议 |
|------|------|------|
| **多级缓存 (L1 Local + L2 Redis)** | 无 | 秒杀 Hot Key（如 `seckill:stock:xxx`）高并发时可能打满 Redis 网卡。建议：引入 go-cache，热点 Key 提升至本地缓存（TTL 5s），降低 Redis 压力。 |
| **Lua 脚本健壮性** | 有风险 | `voucher_script.lua` 中 `tonumber(redis.call("get", stockKey))` 当 key 不存在时返回 nil，`nil <= 0` 可能报错。建议：`local stock = redis.call("get", stockKey) or "0"` 再判断。 |
| **Redis 库存与 MySQL 一致性** | 依赖预热 | 仅 AddSeckillVoucher 时写 Redis。历史数据迁移或 Redis 清空后，库存可能缺失。建议：Lua 中若 stock 不存在，可返回特殊码，由应用层从 MySQL 加载并回填后重试；或启动时批量预热已有秒杀券库存。 |

### 3.3 低优先级

| 项目 | 现状 | 建议 |
|------|------|------|
| **缓存失效** | 秒杀券缓存无主动失效 | 秒杀券信息变更（如改时间）时，缓存未主动删除。当前 TTL 5 分钟可接受，若有运营修改需求可加 Cache-Aside 删除。 |
| **监控与可观测性** | 无 | 可补充：秒杀成功率、Redis 命中率、MQ 堆积、关单量等指标，便于排查与容量规划。 |

## 四、成熟度评估

| 维度 | 评分 | 说明 |
|------|------|------|
| **防超卖** | ⭐⭐⭐⭐⭐ | Lua + MySQL 乐观锁 + 分布式锁，三重保障 |

| **防穿透** | ⭐⭐⭐⭐⭐ | 秒杀券布隆过滤器 + 缓存 |
| **可维护性** | ⭐⭐⭐⭐ | 分层清晰，文档完善 |

**综合**：已补充唯一索引、限流、秒杀券布隆过滤器，方案成熟度较高，可支撑生产环境。

## 五、推荐实施顺序

1. **唯一索引**（低风险，立即生效）
2. **Lua 脚本 nil 处理**（小改动，提高健壮性）
3. **Sentinel 限流**（防雪崩）
4. **秒杀券布隆过滤器**（防穿透）
5. **多级缓存**（高流量场景）
